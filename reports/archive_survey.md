# NISAR archive look-direction census

Generated by `survey_archive.py` (`make survey`). Companion to
[`lookdir_verification.md`](lookdir_verification.md); evidence for
[RAiDER issue #676](https://github.com/dbekaert/RAiDER/issues/676).

## Question

The verification run finds real NISAR GUNWs to be left-looking. Is *any*
real NISAR acquisition right-looking?

## Answer

**No.** Across `NISAR_L2_GUNW_BETA_V1` — 10217 granules, grouped into
221 distinct (track, pass) combinations, one probed per group:

| lookDirection | (track, pass) groups |
|---|---|
| **Left** | 221 |

Right-looking groups found: **0 of 221 readable**.

The same holds outside the interferogram product — sampling other NISAR beta
collections across distinct tracks:

| product | probed | result |
|---|---|---|
| GCOV | 5 | Left×5 |
| GSLC | 5 | Left×5 |
| RSLC | 5 | Left×5 |

## Why this matters for #676

NISAR is a left-looking mission. RAiDER hardcodes `right` for ray-traced runs
(the run config's `look_dir` never reaches `Raytracing`), so the look direction is
wrong for **every** NISAR product, not just some of them.

The one right-looking product in `lookdir_verification.md` is the JPL sample-suite
granule, which is ALOS-1 PALSAR surrogate data rather than a NISAR acquisition. It is
kept as a control: it proves the reader and the geometric check discriminate both
values instead of always reporting `Left`.

## Method

Granules are enumerated via CMR, grouped by (track, pass), and one per group is
probed with a 64 KB-block `fsspec` HTTP handle, so `h5py` reads only the
HDF5 metadata needed for one scalar (~0.26 MB per granule rather than ~2 GB).

Census wall-clock: 6 min.

```bash
make survey
```
